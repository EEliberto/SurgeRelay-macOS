# Apple 原生开发规范

## 核心原则

本项目使用 Swift 和 SwiftUI 开发 Apple 平台原生应用。

实现功能时按以下优先级决策：

1. Apple Human Interface Guidelines
2. Apple Developer Documentation
3. Apple 官方 Sample Code 和 WWDC 内容
4. 项目现有组件和代码结构
5. 维护良好、许可证兼容的 GitHub SwiftUI 实现

目标是符合 Apple 平台的原生组件、布局和交互习惯，而不是通过圆角卡片、渐变、阴影和毛玻璃模仿所谓的“Apple 风格”。

---

## 技术要求

默认使用：

- Swift
- SwiftUI
- Swift Concurrency
- Observation
- Apple 官方 Framework
- SF Symbols
- 系统字体、语义颜色和系统控件

SwiftUI 是默认 UI 方案。

只有 SwiftUI 无法合理实现需求时，才允许使用 AppKit 或 UIKit。使用桥接时，将代码限制在独立的小型组件中，并简要说明原因。

不要为了简单功能引入第三方 UI 框架。

---

## UI 调研规则

以下情况需要先调研再实现：

- 新增主要页面或窗口
- 重构导航或页面结构
- 实现复杂列表、表格、表单、工具栏、弹窗或菜单
- 调整滚动、固定头部、材质、模糊或窗口行为
- 用户要求“更原生”“更像 Apple”或“打磨 UI”
- 不确定 Apple 平台通常如何实现某个组件

调研顺序：

1. 查阅与当前问题直接相关的 Apple HIG 或 SwiftUI API。
2. 检查项目中是否已有相同或相似实现。
3. Apple 官方资料无法解决具体实现问题时，再搜索 GitHub。

不要为了完成调研而大范围搜索。

每次只查与当前问题直接相关的少量资料。找到明确可行的原生方案后立即停止继续搜索。

GitHub 仅用于参考具体 SwiftUI 实现，不得用开源项目的视觉风格取代 Apple 官方规范。

---

## 调研输出

简单改动不需要单独输出调研过程，直接实现。

只有涉及明显 UI 结构选择或存在多种可行方案时，编码前简要说明：

- 准备使用的原生组件或 API
- 为什么选择该方案
- 是否需要参考现有项目或 GitHub 实现

控制在几句话以内，不要输出长篇调研报告。

---

## 原生组件优先

优先使用系统语义化组件，例如：

- `NavigationSplitView`
- `NavigationStack`
- `List`
- `Table`
- `Form`
- `Section`
- `GroupBox`
- `Toolbar`
- `Menu`
- `Settings`
- `Inspector`
- `Popover`
- `Sheet`
- `Alert`
- `ContentUnavailableView`
- `ProgressView`
- `Gauge`
- `Picker`
- `Toggle`
- `Button`
- `TextField`
- `ScrollView`
- `safeAreaInset`
- `containerBackground`

不要使用大量 `VStack`、`RoundedRectangle`、`overlay` 和手工 padding，重新拼装系统已经提供的组件。

---

## 视觉规范

### 字体

优先使用系统语义文本样式，例如：

- `.title`
- `.headline`
- `.body`
- `.callout`
- `.subheadline`
- `.caption`

没有明确理由时，不要硬编码字体、字号和字重。

### 颜色

优先使用：

- `.primary`
- `.secondary`
- `.tertiary`
- `.accentColor`
- 系统语义背景色
- 系统 material 和 separator

不要使用固定 RGB 值模拟系统颜色。

### 图标

优先使用语义正确的 SF Symbols。

图标应与功能含义一致，不要仅根据外观选择，也不要用 emoji 代替正式 UI 图标。

### 装饰

不要默认添加：

- 圆角卡片
- 渐变
- 浓重阴影
- 发光
- 多层透明背景
- 无实际层级意义的毛玻璃

只有内容层级或平台交互确实需要时才使用。

---

## 布局规则

优先使用 SwiftUI 自适应布局。

避免：

- 大量固定宽高
- 负 padding
- 无原因的 `offset`
- 用 `GeometryReader` 修复普通布局
- 用空白 `Spacer` 强行定位
- 为单一截图写死窗口尺寸
- 为特定文字长度专门调整布局

固定尺寸仅适用于图标、缩略图或有明确规格的控件。

---

## macOS 交互

macOS 界面应考虑：

- 窗口尺寸变化
- Hover
- 键盘操作
- Esc 和 Return 行为
- 焦点顺序
- 快捷键
- 右键菜单
- 工具栏和菜单命令
- 菜单栏窗口的失焦行为

不要直接套用移动端页面和交互模式。

---

## 状态和代码结构

优先使用 SwiftUI 原生数据流：

- `@State`
- `@Binding`
- `@Environment`
- `@Observable`
- `@Bindable`

避免：

- 为简单页面创建复杂 ViewModel
- 将所有状态放入一个全局对象
- 在 `body` 中执行副作用
- 重复保存可推导状态
- 为了抽象而创建无意义的 protocol
- 创建语义模糊的通用 UI 组件

优先修改现有组件，不要在已有实现旁边重新创建一套相似组件。

---

## 第三方依赖

默认不引入新的第三方依赖。

只有 Apple 官方 API 无法合理实现需求时才考虑依赖，并简要说明：

- 官方 API 为什么不足
- 依赖解决什么问题
- 是否仍在维护
- 许可证是否兼容

不要为了一个布局、动画、颜色或简单控件引入完整 UI 框架。

---

## Build 与验证

默认只执行完成本次修改所必需的 Build。

要求：

1. 确认项目能够成功编译。
2. 修复本次修改引入的编译错误。
3. 不主动运行单元测试、UI 测试或完整测试套件。
4. 不主动启动 App 进行人工界面测试。
5. 不为了验证 UI 创建额外测试代码。
6. 不执行耗时较长的清理、归档或全平台构建。
7. 用户明确要求时，才运行额外测试或验证。

如果项目本身存在与本次修改无关的编译错误，应明确指出，不要扩大修改范围。

---

## Token 与上下文控制

始终尽量节省 token 和上下文。

### 阅读代码

- 先定位相关文件，再读取必要范围。
- 不要默认读取整个仓库或完整大文件。
- 读取大文件前先检查大小或行数。
- 优先使用 `rg`、`find`、`sed -n` 等定位具体代码。
- 命令输出可能较大时，应限制输出范围。
- 不要重复读取刚刚已经读取过的内容。
- 不要读取与当前任务无关的文件。

### 搜索资料

- 只搜索与当前 UI 问题直接相关的关键词。
- Apple 官方资料通常查看 1–3 个直接相关页面即可。
- GitHub 仅在官方资料无法回答实现问题时使用。
- 通常比较 1–3 个相关实现即可。
- 找到可靠方案后立即停止继续搜索。
- 不要输出大段文档摘要。

### 修改代码

- 优先进行最小范围修改。
- 不要无关重构。
- 不要顺便格式化大量未修改代码。
- 不要创建没有实际需要的抽象或文件。
- 不要为了展示过程输出完整文件内容。

### 回复

完成后只需简要说明：

- 修改了什么
- 使用了哪些关键原生 API 或组件
- Build 是否通过
- 是否存在需要用户自行检查的问题

除非用户要求，不要输出长篇工作日志、完整调研报告或逐文件解释。

---

## 禁止事项

除非存在明确依据，否则禁止：

- 把 Apple 风格理解为圆角卡片、渐变、阴影或毛玻璃
- 手工重建系统已有控件
- 使用固定颜色模拟系统颜色
- 使用大量固定尺寸、负 padding 或 offset
- 引入不必要的第三方 UI 库
- 未经说明使用 AppKit/UIKit
- 臆造 SwiftUI API
- 未实际查阅资料却声称符合 Apple HIG
- 自动运行测试套件
- 自动进行与当前任务无关的重构
- 为了“更加完善”扩大任务范围

---

## 默认工作流程

1. 阅读当前任务和相关项目规则。
2. 定位与任务直接相关的文件。
3. 检查项目现有组件和实现。
4. UI 决策不明确时，进行有限的 Apple 官方资料调研。
5. 官方资料不足时，再有限搜索 GitHub 实现。
6. 采用最原生、最简单、修改范围最小的方案。
7. 执行必要的 Build。
8. 修复本次修改引入的编译错误。
9. 简要汇报结果。

不要默认运行测试。功能和视觉效果由用户自行测试。
